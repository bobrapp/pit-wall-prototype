## Design Principles for v4.0

### 1. Simpler
The Workbench is no longer the mandatory entry point. Every action a team member needs — confirm, override, challenge, evaluate — is available where they already work. No new login. No new app to learn.

### 2. Safer
- Data quality evaluation is first-class, not a footnote. Every forecast carries a 6-dimension DQ score visible on every surface.
- HIBT (How-I-Built-This) is treated as an immutable audit log from day one — not retrofitted later.
- Second Opinion auto-trigger blocks Gate 3 commits without requiring Workbench access.

### 3. Deeper Data Quality
v2.4 had source freshness flags. v4.0 has a full 6-dimension DQ engine: freshness, lineage completeness, override accuracy history, cross-surface consistency, replay determinism, and signal coverage.

## Data Quality Engine — v4.0 Specification

### DQ Score Formula

```
DQ_score = w1 * freshness + w2 * lineage + w3 * override_accuracy
         + w4 * consistency + w5 * replay + w6 * signal_coverage
```

| Dimension | Weight | Inputs | Computation |
|---|---|---|---|
| `freshness` | 25% | Feed timestamps, staleness thresholds per source | Weighted avg of (1 - age/threshold) per source, capped at 0 |
| `lineage` | 20% | HIBT lineage graph coverage | % of forecast value traceable to versioned, hashed source |
| `override_accuracy` | 20% | HIBT historical overrides, actuals | Rolling 6-cycle MAPE for this analyst × nameplate × segment |
| `consistency` | 15% | Cross-surface sync timestamps + values | % of surfaces showing current primary value |
| `replay` | 10% | HIBT replay run vs original | % match on most recent 3 replay runs |
| `signal_coverage` | 10% | Signal catalog completeness | % of catalog signals with a non-stale reading |

**Score thresholds:**
- 85-100: Green — trusted for Gate commit
- 70-84: Yellow — commit allowed with warning logged
- 55-69: Red — GovOps review required before commit
- <55: Block — hard gate, escalation required

### Staleness Thresholds by Source

| Source | Yellow | Red | Block |
|---|---|---|---|
| Dealer Pulse (regional) | 3 days | 7 days | 14 days |
| Tariff feeds | 2 days | 5 days | 10 days |
| MIA Competitive Scan | 3 days | 7 days | 14 days |
| Macro Consumer Index | 7 days | 14 days | 30 days |
| Fleet Demand Signal | 5 days | 10 days | 21 days |
| Incentive Elasticity Model | 7 days | 14 days | 30 days |

## Surface Adapter Contracts (v4.0)

Each surface adapter implements a standard interface:

```python
class SurfaceAdapter(ABC):
    @abstractmethod
    async def push_forecast_digest(self, user: User, cycle: Cycle) -> None:
        """Push daily/weekly digest in surface-native format"""

    @abstractmethod
    async def push_challenge(self, challenge: SOChallenge, sme: User) -> str:
        """Push second opinion challenge; return surface-native message ID"""

    @abstractmethod
    async def handle_action(self, action: UserAction) -> ActionResult:
        """Handle confirm/override/dismiss from surface-native payload"""

    @abstractmethod
    async def push_notification(self, notif: Notification, user: User) -> None:
        """Push point notification (DQ alert, championship update, etc.)"""
```

## Personal Delighters — v4.0

### My Dashboard
Every analyst gets a personal homepage:
- My accuracy this cycle vs. prior cycles (sparkline)
- My overrides: which beat the primary, which didn't (Ghost Lap personal view)
- My season ranking + points trajectory
- My pending actions (3 max, highest priority first)
- Signals most relevant to my nameplates

### Morning Digest (personalized per surface)
- **Outlook version:** HTML email with inline Adaptive Cards, clickable confirm/override buttons, DQ score table, one personal insight
- **Slack version:** Block kit message in preferred channel, `/pitwall` quick commands, personal ranking update
- **Teams version:** Notification + adaptive card in My Activity feed
- **Glean version:** Answer card surfaced in Glean home feed, priority ranked

### Ghost Lap Personal Accuracy Card
- Your override MAPE vs. primary model MAPE by nameplate
- Cycles where your call beat the AI, and by how much
- Running season accuracy score (feeds Championship formula)
- Trend line: are you getting better?

### Championship Delighters
- Points notification when you move up (Slack DM + Outlook + Glean simultaneously)
- Lap Clap receipt: animated notification when a teammate claps for you
- Season close ceremony invitation (calendar invite, PPT summary auto-generated)
- Grand Prix PPT deck: your highlights, top moments, accuracy story

### "You Were Right" Moment
When actuals confirm an override was better than the primary model:
- Push on all preferred surfaces: "Your May Silverado HD call was right. MAPE 2.8% vs 5.5% primary. 🏁 +12 Championship points"
- Logged to HIBT as positive reinforcement signal

## End-to-End Conversation Architecture

### Conversation Lifecycle
```
1. INITIATION — any surface
   User: question, command, or proactive push from system

2. CONTEXT ASSEMBLY — Pit Wall AI backend
   - Load user profile (persona, preferred surface, current cycle assignments)
   - Load relevant forecast state (nameplate, DQ score, pending actions)
   - Load HIBT context (last override, ghost lap history, active challenges)
   - Load signal feed (dealer pulse, tariff, competitive for relevant nameplates)

3. RESPONSE GENERATION
   - Generate surface-native response (Slack block kit / Outlook HTML / Teams adaptive card)
   - Include inline action options appropriate to user's seat
   - Surface DQ warnings inline if relevant

4. ACTION HANDLING
   - User confirms/overrides/dismisses within their surface
   - Action written to HIBT ledger before any state change
   - Canonical value store updated (Unity Catalog)
   - All surface caches invalidated (consistency sync triggered)

5. CONFIRMATION + FOLLOW-UP
   - Confirmation sent back on originating surface
   - Cross-surface sync status checked
   - Next proactive prompt queued
```

### Conversation State Machine
```
STATES: idle → listening → assembling_context → generating → awaiting_action → complete
TRANSITIONS:
  idle → listening: user message OR system proactive push
  listening → assembling_context: message parsed, intent confirmed
  assembling_context → generating: HIBT + forecast context loaded
  generating → awaiting_action: response sent, action options presented
  awaiting_action → complete: user action received + logged
  awaiting_action → idle: timeout (14 min) — reminder sent
  complete → idle: follow-up queued for next morning digest
```
